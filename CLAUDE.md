# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A side-scrolling run-and-gun browser game built in **vanilla JavaScript** with the HTML5 Canvas 2D API and the Web Audio API. No build step, no frameworks, no package manager, no dependencies. The entire game runs from `index.html` loading a fixed sequence of plain `<script>` files (no modules/imports — everything shares the global scope).

> **BLANK BASE STATUS:** This repo was intentionally gutted into a **reusable jump'n'run starter base**. **All gameplay mechanics are kept and fully functional** (physics, climbing, swimming, jetpack, weapons, enemies, bosses, particles, audio, sprites, input, HUD), but the original "Bad Mario" level content and brand identity were stripped. There is now **one playable demo level per mode**: Mode A (NORMAL/procedural, `loadBlueprint`+`buildModule` in [level.js](level.js)) and Mode B (CLASSIC/hand-placed, `build_demo` in [classic.js](classic.js)). The enemy/boss/weapon classes all still exist and can be spawned — only the level layouts and Bad-Mario branding texts were removed. **Add your own levels** in those two builders; **change the look** via `CONFIG.LEVELS` palettes/`DECOR` ([config.js](config.js)), branding text ([index.html](index.html)), and effects ([style.css](style.css) + `draw*` in [game.js](game.js)). The descriptions below still document mechanics that remain in place; ignore references to specific old levels/themes as content.

## Running & testing

There is no build/lint/test tooling. To run the game, open `index.html` in a Chromium browser (Chrome/Edge recommended for Web Audio). Audio requires a user gesture (click) before it plays.

`node` and `git` are typically **not on PATH** in this environment. To verify a change works, run the game headless via Edge against a temporary diagnostic HTML page that loads the scripts and exercises the logic (this is the established verification method here — see the `verify-with-headless-edge` memory). The canvas needs a real browser; there is no Node harness.

## Architecture

Scripts load in dependency order (see [index.html](index.html) bottom): `config` → `audio` → `input` → `particles` → `sprites` → `entities` → `enemies` → `player` → `level` → `classic` → `game`. Each file defines globals (classes or singletons) consumed by later files. `game.js`'s `window.onload` is the single entry point: it calls `Assets.init()` then `new Game().start()`.

Key globals and their roles:

- **`CONFIG`** ([config.js](config.js)) — physics constants, per-level theming (`LEVELS[1..10]`, palette + `DECOR` name), and shared colors. 10 levels grouped into 5 visual themes (theme = `ceil(level/2)`).
- **`Game`** ([game.js](game.js)) — the orchestrator. Owns the fixed-timestep `loop()` → `update(dt)` → `draw()` cycle, the camera, the `state` machine (`MENU`/`PLAYING`/`PAUSED`/`GAMEOVER`), the projectile array, screen shake, combo system, HUD/DOM sync, and **all collision resolution** (projectile↔enemy/platform/player, player↔item/enemy, boss headshot logic). When adding gameplay interactions, this is usually where they wire together. Sets `window.gameInstance = this` each frame.
- **`Entity`** ([entities.js](entities.js)) — base class with AABB `checkCollision`. Subclasses: `Platform` (carries all terrain gimmick flags — `isBouncy`, `isCrumbling`, `isSpiky`, `isMoving`, `isFireTrap`, `isSolidGround`), `Ladder`, `Collectible`, `Projectile`, `Corpse`.
- **`Player`** ([player.js](player.js)) — extends `Entity`. Movement/climbing/physics, the inventory (`{weaponName: ammo}`, `BAT` has `Infinity`), and the large per-weapon `update()` firing logic + `draw()` rendering switch. `ammo` is a getter over `inventory[weapon]`. The character is drawn **fully procedurally** (`drawMarioBody`/`_drawHead`, poses for idle/walk/crouch/air) plus the procedural weapon-arm — it no longer uses the `Assets.playerWalk` sprite sheet. `draw()` applies `vis = standH/140` scale + a foot-alignment translate so the shoes sit exactly on the hitbox bottom at any size (Classic shrinks the player via `standH` in `Game.startPlay`). **Swimming**: when the active level sets `LevelGenerator.waterY` (water surface y; null = none) and the player's centre is below it, `update()` switches to swim physics (gentle `SWIM_GRAVITY` sink, jump key = upward stroke `SWIM_STROKE`, slower horizontal); `Game.drawWater()` renders the tint/surface/bubbles and `drawClassicBackground` paints a blue underwater gradient. Classic 2-2 (`build_2_2`) is the swim level. **Jetpack** (`JETPACK` power-up item): pickup sets `hasJetpack`+`jetpackFuel`; holding the jump key *in the air* thrusts up (`JETPACK_THRUST`, capped at `JETPACK_MAX_RISE`) until fuel runs out, refuelling on the ground (`JETPACK_REFUEL`); drawn as a backpack + jet flame with a fuel bar over the head. All jetpack/swim tunables live in `CONFIG`. The `JETPACK` item has its own sprite in `SpriteGenerator`/`Assets.init`, is dispensed by Classic `?`-blocks (in `C_POWERUPS`) and Story modules (`ARSENAL`/`BOUNCE`), and is handled in `Game.update`'s item-pickup block.
- **Enemies** ([enemies.js](enemies.js)) — `Enemy` base + concrete types (`ZombieEnemy` with `enemyType` variants, `SoldierEnemy`, `SpiderEnemy`, `DemonEnemy`, `TridentDemonEnemy`, `HellhoundEnemy`, `BloaterEnemy`, `GiantZombieEnemy`) and three bosses (`BossGolem`, `BossMech`, `BossHell`). Each owns its AI in `update(dt, game)` and visuals in `draw()`.
- **`LevelGenerator`** ([level.js](level.js)) — procedural level builder. Levels are **module blueprints**: `loadBlueprint(level)` returns an ordered list of module names (`RUN`, `GAP`, `TOWER`, `HORDE`, `BOSS`, `EXIT`, etc.); `buildModule()` is a big switch that appends platforms/ladders/enemies/items at a moving `cursorX`, generated lazily as the camera advances. Boss levels are even (2,4,6,8,10) and end with a `BOSS` module; odd levels end with an `EXIT` flag (`goalX`). `spawnThemeEnemy(x, lvl, diff, role)` maps an abstract role (`basic`/`fast`/`tank`/`ranged`/`special`/`flyer`) to the concrete enemy type appropriate for the level's theme.
- **`CLASSIC_AVAILABLE` + `LevelGenerator.prototype.buildClassic`** ([classic.js](classic.js)) — the **CLASSIC mode**: hand-authored, originalgetreue Super-Mario levels (currently World 1-1) built at absolute coordinates instead of procedurally. Loaded *after* level.js (extends the prototype). Toggled by the start-menu switch (`Game.setMode`, persisted as `badMarioMode`); `Game.classicMode` flows into `LevelGenerator.classicMode`, which makes `update()` call `buildClassic()` once per (re)build and skip the blueprint loop. Same physics/mechanics as story mode. Tile grid `C_TILE=92` (sized so the 80px player is ~1 tile wide, SMB ratio); pipe heights are a separate px map (`C_PIPE_H`, max ~175px vs ~200px jump apex) so they stay jumpable regardless of tile size. Blocks (`?`/brick) are **solid + `bumpable`**: a head-bump from below (handled in `Player.handleCollisions` vy<0 branch) — or a player projectile hit (in `Game`'s projectile↔platform loop) — calls `Game.bumpBlock`/`dispense`. `?` blocks pop their `content` (COIN instant, or a `Collectible` for HEART/STAR/weapons spawned at a reachable height) and become a solid `USED` block; bricks with content dispense it, empty bricks shatter (removed from `platforms`). Pipes/stairs are plain solid (`isSolidGround`). Enemies are the procedurally-drawn **`GoombaEnemy`/`KoopaEnemy`** (rancid Mario originals, base `ClassicWalker` in [enemies.js](enemies.js) — patrol with wall+ledge turning, stompable, lightweight non-Corpse death), spawned by `cEnemy`. **`KoopaEnemy` has a shell mechanic** (`shellState` walk→shell→slide): first hit (stomp/shot) retracts it to an idle shell; hitting/kicking an idle shell sends it sliding fast (mows down other enemies); a sliding shell is destroyed by another hit — wired in `KoopaEnemy.takeDamage`/`kick`/`update` + the player↔enemy block in [game.js](game.js) (`isShellAny`/`isIdleShell`). **32 classic levels** (8 worlds × 4): **All 32 levels (`1-1`–`8-4`) are hand-authored** (`build_1_1`…`build_8_4`, dispatched per level in `buildClassic`); `build_world(world, sub, diff)` remains only as an unused safety-net fallback. Swim levels: **2-2**, **7-2** (both set `waterY` + drop a `JETPACK`). Completing **8-4** (Bowser defeated, no level 33) triggers the victory screen. `build_5_2`/`build_6_2`/`build_8_2` override `classicTheme` to `over` at their top (the sub→theme guess says `under` but those stages are overworld). **Bullet-Bill cannons** (`cCannon`): a solid `style:'CANNON'` platform with `isCannon` whose `Platform.update` fires `BULLET` projectiles at the player via `window.gameInstance` (same pattern as `isFireTrap`); used heavily in 5-1. A build function may override `this.classicTheme`/`classicUnder` at its top to correct the sub→theme guess (e.g. `build_5_2` forces `over`). Hand-authored specials: **2-2** is a swim level (`waterY` set; SMB's underwater stage adapted since there's no real fish/blooper AI); **3-3** is the athletic lift level (`cLift` = vertically pendling `isMoving` non-solid platform over a bottomless pit); world-3 over-levels are night (`classicNight` → black sky in `drawClassicBackground`). `cLift`/`cPlat`/`cStair` are the platform helpers; `cHammer`/`cBullet`/`cBowser`/`cPiranha` the special enemies. `CLASSIC_AVAILABLE`/`CLASSIC_LABELS`/`CLASSIC_THEMES` are filled by a loop (level N → world `ceil(N/4)`, sub `((N-1)%4)+1`; sub 1/3 = over, 2 = under, 4 = castle). Completing `8-4` triggers victory. Castle hazards use **Piranha plants** (`PiranhaPlantEnemy` via `cPiranha` — emerge from pipes, not stompable/`noStomp`, shoot to kill) — the old fire-trap columns were removed (they spammed projectiles and caused lag). Later worlds get tougher enemies via `worldHpMul`. Each platform carries `p.ctheme` (`'over'`/`'under'`/`'castle'`) → palette in `Platform.drawClassic`; the theme also drives the background and lava (castle only) in [game.js](game.js). Build helpers: `cFloor/cPipe/cBlock/cStair/cPlat/cDrink/cEnemy/cHidden`. **Hidden blocks** (`cHidden`, style `'HIDDEN'`) are invisible until bumped, then reveal as `USED`. `cEnemy` scales **count** by difficulty (princess fewer, badass more); enemy **HP** scales via `LevelGenerator.spawn` (+ `worldHpMul` for later worlds). Enemy roster ([enemies.js](enemies.js)): `GoombaEnemy`, `KoopaEnemy` (shell), `ParatroopaEnemy` (winged → loses wings on first hit), `PiranhaPlantEnemy`, `HammerBroEnemy` (hops + throws hammers), `BulletBillEnemy` (straight flyer), and `BowserEnemy` (castle boss: `isBoss`+`noStomp`, breathes fire, only killable by shots — death triggers `handleBossDefeat`→advance). `build_world` is a weighted **segment** generator (run/gap/pipes/bricks/stairs/pyramid/coins/floaters) varied per theme so levels don't repeat; weapon blocks pull from `cWeapon(world)` (full arsenal, weighted by world). Every castle (X-4) ends with Bowser. Each level has its own darker/minor chiptune (`AudioManager.playChiptune` keys `'1-1'..'1-4'`/`'BOSS'`). Per-mode level unlock is tracked separately (`badMarioClassicUnlocked`), shown in the start-menu level selection.

**Characters** ([config.js](config.js) `CONFIG.CHARACTERS`: MARIO/LUIGI/SONIC) are chosen via the start-menu switch (`Game.setCharacter`, persisted `badMarioChar`) and passed to `new Player(x,y,charKey)`. Each defines `jump`/`speed`/`dmg` (damage taken)/`airJumps` (double-jump) + palette + `hat` (`'cap'`|`'spikes'`); `Player` applies these to physics/`takeDamage` and `drawMarioBody`/`_drawHead` swap colors and cap-vs-spikes. Luigi: higher jump, takes more damage; Sonic: fast + double jump + spiky. The Classic player is scaled smaller in `Game.startPlay` (`standH`≈0.7×) and `Player.draw` applies a `vis = standH/140` scale + foot-alignment translate so feet sit exactly on the ground line. Rendering: platforms carry a `.style` (`GROUND`/`PIPE`/`BRICK`/`QUESTION`/`USED`/`STAIR`) handled by `Platform.drawClassic` ([entities.js](entities.js)) with a small bump-hop via `bumpTimer`; `Game` uses `CONFIG.CLASSIC` palette + `drawClassicBackground`/`drawClassicCastle` and skips lava. The PNG references for all original levels live in [levels PNGs/](levels%20PNGs/).
- **`SpriteGenerator` / `Assets`** ([sprites.js](sprites.js)) — **all graphics are drawn procedurally at runtime** into offscreen canvases (no image asset files except the background JPG). `SpriteGenerator.generate(type, variant, level)` produces an 8-frame sprite sheet. `Assets.init()` pre-renders every player/item/enemy/boss sprite once into the `Assets` singleton; entity `draw()` methods blit frames from there.
- **`AudioManager`** ([audio.js](audio.js)) — hybrid engine: MP3 samples (in `sound files/`, loaded via fetch with the space URL-encoded as `%20`) for music and weapon SFX, plus procedurally synthesized Web Audio sounds. Handles dynamic BGM that swaps to the boss "thrash" track when a boss is active, and sustained loop sounds (minigun/flamethrower/chainsaw) ticked via `tickSustains()`. **`audioTheme`** (`'METAL'` default | `'CLASSIC'`, set by the start-menu sound switch via `Game.setAudioMode`, persisted as `badMarioAudio`) selects only the **BGM**: `CLASSIC` replaces the MP3 music with an 8-bit chiptune (`playChiptune` OVERWORLD/BOSS loops scheduled via `setTimeout`+`chipNote`). All weapon/effect SFX stay the same in both themes. The theme is orthogonal to the level mode — both Story and Classic levels can use either music.
- **Menu/UI** ([index.html](index.html) + [style.css](style.css)) — the start menu is a translucent panel of labelled `.menu-section` rows (Modus/Sound/Charakter/Level/Schwierigkeit) over the `bad-mario-bg.jpg` overlay (kept semi-transparent so the background art shows through); Layout is always the two edge columns (`.menu-cols` absolute at `left:0`/`right:0`) with the centre free for the background art and the title bar pinned to the bottom. The whole `#game-container` (menu + game) is force-rotated to landscape on phones via the portrait `transform: rotate(90deg)` hack, so menu/game always run landscape (hold the phone sideways). Because the container is rotated, menu column sizes use `vmax`/`vmin` (not `vw`/`vh`) so they track the rotated container's real width/height. The on-screen touch controls (`#mobile-controls`) and `#zoom-controls` are gated by body classes: `Game.loop` toggles `body.in-game` (PLAYING/PAUSED) and `Game.resize` toggles `body.touch` (touch device / ≤950px), and CSS shows the D-pad/buttons only when `body.in-game.touch`. So they no longer appear in menus or on desktop.
- **`InputHandler`** ([input.js](input.js)) — unifies keyboard, on-screen mobile D-pad/buttons, and Gamepad API into a keycode-based `isDown`/`isJustPressed`/`isJustReleased` interface. Gamepad menu/pause actions dispatch custom `gamepadStart`/`gamepadPause` window events (simulated KeyboardEvents are unreliable). Exposes itself as `window.inputHandlerRef` for diagonal aiming.
- **`ParticleManager`** ([particles.js](particles.js)) — blood, explosions, fire, casings, level-up effects.

## Conventions & gotchas

- **Code comments are in German.** Match that when editing existing files.
- Everything is global scope — no `import`/`export`. A new class/file must be added as a `<script>` in [index.html](index.html) in correct dependency order.
- **`src/levels/level1.js` is orphaned/dead code** — it is *not* referenced by `index.html` and defines an alternate module architecture (`Level1`, `PlayerModule`, etc.) that the live game does not use. The real level system is the `LevelGenerator` blueprint approach in [level.js](level.js). Don't confuse the two.
- Persistent state uses `localStorage`: `badMarioHighscore` and `badMarioUnlockedLevel`.
- Adding a weapon touches several places in lockstep: the pickup ammo amount in `Game.update`'s item-collision block ([game.js](game.js)), the firing branch + cooldown + draw branch in [player.js](player.js), the sprite case in `SpriteGenerator`/`Assets.init`, and usually an SFX in [audio.js](audio.js).
- Difficulty (`princess`/`regular`/`badass`) scales enemy HP in `LevelGenerator.spawn()`.
